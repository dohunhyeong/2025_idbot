> **평가 일자**: 2026-03-17
> **평가 대상**: 법정감염병 챗봇 RAG 시스템 3단계 발전 과정
> **평가 방법**: LLM-as-Judge (GPT-4o)
> **평가 샘플**: 100개 질문 (RAG_qa.xlsx 전체 698개 중 random_state=42로 추출)

## 1. 평가 개요

### 1.1 평가 목적

법정감염병 챗봇의 RAG 시스템이 세 단계를 거쳐 발전하면서 답변 품질이 얼마나 향상되었는지 정량적으로 측정한다.
단순 체감 비교가 아닌 **LLM-as-Judge** 방식을 채택하여 참조 정답 없이도 객관적인 수치를 확보한다.

### 1.2 평가 대상 시스템

| 구분 | 명칭 | FAISS DB | RAG 방식 | LLM | Embedding |
|------|------|----------|----------|-----|-----------|
| **S1** | 전처리X + Naive | `2024_proj_openai/data/faiss_index` | Simple chain (k=2) | gpt-3.5-turbo | text-embedding-3-large |
| **S2** | 전처리O + Naive | `db/faiss_total/index` | Simple chain (k=2) | gpt-3.5-turbo | text-embedding-3-large |
| **S3** | 전처리O + Advanced | `2025_proj(hdh)/resources/vectorstore/` (12개 도메인) | Intent→Route→Retrieve→Summarize | exaone3.5:7.8b (Ollama) | bge-m3 (Ollama) |

**단계별 변경 포인트**

- **S1 → S2**: 문서 전처리 적용 (동일한 Naive RAG 체인 유지)
- **S2 → S3**: RAG 아키텍처 고도화 (Intent 분류 → 도메인 라우팅 → 검색 → 요약), 벡터 DB 세분화(12개), LLM·Embedding 교체

## 2. 평가 방법론

### 2.1 데이터 수집

- **S1 답변**: `RAG_qa.xlsx`의 `answer` 컬럼에서 직접 로드 (별도 API 호출 불필요)
- **S2 답변**: `http://localhost:8002/query` 엔드포인트 호출 (concurrency=5, timeout=30s)
- **S3 답변**: `http://localhost:8000/query` 엔드포인트 호출 (concurrency=2, timeout=120s)
- **수집 결과**: S1=100/100, S2=94/100 (6건 오류), S3=100/100

### 2.2 평가 방식

두 가지 방식을 병행하여 상호 보완적으로 신뢰도를 높였다.

#### (1) 절대 평가 (Absolute Scoring)

참조 정답 없이 **질문–답변 쌍만으로** 3개 지표를 각각 1~5점으로 채점.

| 지표 | 설명 |
|------|------|
| **relevance** (관련성) | 답변이 질문에 직접 대답하는가 |
| **completeness** (완전성) | 질문이 필요로 하는 정보를 충분히 포함하는가 |
| **helpfulness** (유용성) | 사용자에게 실질적으로 도움이 되는 답변인가 |
| **total_score** | 위 3개 지표의 평균 |

**프롬프트 구조**:
```
당신은 법정감염병 챗봇의 답변 품질을 평가하는 전문가입니다.
참조 답변 없이 질문과 답변만으로 평가합니다.

[평가 기준]
- relevance: 답변이 질문에 직접 대답하는가 (1~5)
- completeness: 질문이 필요로 하는 정보를 충분히 포함하는가 (1~5)
- helpfulness: 사용자에게 실질적으로 도움이 되는 답변인가 (1~5)

[질문]: {question}
[답변]: {answer}

JSON만 출력: {"relevance": <1-5>, "completeness": <1-5>, "helpfulness": <1-5>, "reason": "<근거>"}
```

- **모델**: GPT-4o (`temperature=0`, `response_format=json_object`)
- **총 호출 수**: 100개 × 3시스템 = **300회**

#### (2) Pairwise 비교 (Pairwise Comparison)

동일 질문에 대한 두 시스템의 답변을 제시하고 **어느 쪽이 더 나은지 직접 비교**.
절대 평가보다 신뢰도가 높으며 미세한 차이도 포착 가능.

**비교 쌍**: S1 vs S2 / S2 vs S3 / S1 vs S3

**프롬프트 구조**:
```
당신은 법정감염병 챗봇의 답변 품질을 비교하는 전문가입니다.
같은 질문에 대한 두 시스템의 답변을 비교하여 어느 것이 더 나은지 판단하세요.

[질문]: {question}
[답변 A]: {answer_a}
[답변 B]: {answer_b}

판단 기준: 정확성, 관련성, 완전성, 유용성을 종합적으로 고려합니다.

JSON만 출력: {"winner": "A" 또는 "B" 또는 "tie", "reason": "<근거>"}
```

- **총 호출 수**: 100개 × 3쌍 = **300회**

## 3. 평가 결과

### 3.1 절대 평가 결과

#### 시스템별 평균 점수

| 시스템 | relevance | completeness | helpfulness | **total_score** |
|--------|-----------|--------------|-------------|-----------------|
| S1 (전처리X + Naive) | 4.53 ± 0.904 | 4.06 ± 1.071 | 4.08 ± 1.051 | **4.223 ± 0.981** |
| S2 (전처리O + Naive) | 4.60 ± 0.853 | 4.12 ± 0.967 | 4.10 ± 0.959 | **4.273 ± 0.894** |
| S3 (전처리O + Advanced) | 4.77 ± 0.617 | 4.68 ± 0.803 | 4.69 ± 0.775 | **4.713 ± 0.723** |

#### 단계별 개선율

| 지표 | S1→S2 (전처리 효과) | S2→S3 (Advanced RAG 효과) | S1→S3 (전체 개선) |
|------|--------------------|--------------------------|--------------------|
| relevance | +1.5% | +3.7% | **+5.3%** |
| completeness | +1.5% | +13.6% | **+15.3%** |
| helpfulness | +0.5% | +14.4% | **+15.0%** |
| **total_score** | **+1.2%** | **+10.3%** | **+11.6%** |

#### 주요 인사이트

1. **전처리 효과(S1→S2)는 제한적**: total_score 기준 +1.2% 향상에 그침. 동일한 Naive RAG 체인을 사용하기 때문에 전처리 품질 향상만으로는 큰 차이가 없다.

2. **Advanced RAG 아키텍처의 효과(S2→S3)가 압도적**: total_score +10.3%, 특히 completeness(+13.6%)와 helpfulness(+14.4%)가 크게 향상. Intent 라우팅으로 도메인별 최적 검색이 가능해지고, Summarize 단계가 답변의 완결성을 끌어올린 것으로 분석.

3. **S3의 분산 감소**: std가 S1(0.981) → S3(0.723)으로 줄어들어 **답변 품질의 일관성**도 향상됨.

4. **relevance는 이미 높음**: S1도 4.53점으로 관련성은 기본적으로 확보되어 있었음. 개선의 핵심은 completeness와 helpfulness임.

### 3.2 Pairwise 비교 결과

| 비교 쌍 | A 승리 | B 승리 | 동점 | 유효 n |
|---------|--------|--------|------|--------|
| S1 vs S2 | 33 (33%) | 57 (57%) | 10 (10%) | 100 |
| S2 vs S3 | 7 (7%) | 93 (93%) | 0 (0%) | 100 |
| S1 vs S3 | 9 (9%) | 91 (91%) | 0 (0%) | 100 |

#### 주요 인사이트

1. **S3의 압도적 우위**: S2 대비 93%, S1 대비 91% 승률. Pairwise 비교에서 동점이 0건으로 거의 모든 케이스에서 명확하게 우수.

2. **S1 vs S2는 상대적으로 경쟁적**: S2가 57%로 우위이지만 S1도 33%를 가져감. 절대 평가의 +1.2% 차이와 일관된 결과.

3. **S3가 패배한 9개 케이스 분석**: 주로 S3가 직접적인 단답형 질문에 과도하게 풍부한 답변을 제공하여 핵심 정보가 묻히거나, 간혹 사실 오류(예: 주혈흡충증 감염병 등급 오답)가 발생한 경우.

### 3.3 S3 패배 케이스 상세 분석

S1이 S3를 이긴 9개 질문 중 대표 사례:

| 질문 | 패배 원인 |
|------|-----------|
| 주혈흡충증는 몇급감염병이야? | S3가 4급으로 오답 (정답: 3급). S1이 정확히 3급으로 답변 |
| 사람 보카바이러스는 몇급 감염병이야? | S3가 질문에 직접 답하지 않고 우회적 설명. S1은 제4급으로 명확히 답변 |
| 악구충증 환자관리에 대해 알려줘 | S3가 환자 관리 지침 대신 일반 정보 위주로 답변 |
| 살모넬라균의 전파기간은? | S3가 전파기간 외 불필요한 정보를 많이 포함 |
| 리노바이러스의 관련지침에 대해 몇가지 알려줘 | S1이 표준주의·접촉주의·비말주의 등 구체적 지침 제시 |

→ **S3의 약점**: 단답형·수치 확인 질문에서 간혹 불필요한 정보가 핵심 답변을 희석하거나, 검색된 문서의 정보가 부정확할 때 사실 오류 발생.

## 4. 응답 시간 비교

> S1은 xlsx에서 답변을 로드하여 latency 측정 불가. S2·S3만 비교.

| 시스템 | 평균 응답 시간 | 특이사항 |

- **S3는 S2 대비 약 5.5배 느림**. 로컬 Ollama 모델(exaone3.5:7.8b) 사용과 Intent→Route→Retrieve→Summarize 4단계 파이프라인이 주요 원인.
- S3 일부 케이스는 100초를 초과하기도 함. 실서비스 배포 시 응답 속도 개선이 필요한 과제.

## 5. 종합 결론

### 5.1 성능 요약

```
품질 순서: S3 >> S2 > S1

S3 total_score: 4.713 (+11.6% vs S1)
Pairwise 승률: S3 vs S1 = 91%, S3 vs S2 = 93%
```

### 5.2 전처리의 역할 재평가

전처리(S1→S2)가 단독으로는 미미한 효과(+1.2%)를 보이지만, Advanced RAG와 결합될 때 시너지를 낸다. 전처리된 고품질 문서가 도메인별 벡터 DB(12개)에 적재됨으로써 S3의 Intent 라우팅 효과를 극대화한다.

### 5.3 Advanced RAG 아키텍처의 가치

| 요소 | 효과 |
|------|------|
| Intent 분류 | 질문 유형(증상/예방/신고/지침 등)을 파악하여 최적 도메인으로 라우팅 |
| 도메인별 벡터 DB (12개) | 검색 노이즈 감소, 관련 문서 정밀도 향상 |
| Summarize 단계 | 검색된 문서를 질문 맥락에 맞게 재구성, completeness/helpfulness 향상 |

### 5.4 개선 방향

1. **응답 속도**: S3의 평균 52초는 실서비스에 부적합. Ollama 모델 캐싱, 비동기 처리 최적화, 또는 경량 모델 검토 필요.
2. **사실 정확성**: 감염병 등급, 수치 등 단답형 정보에서 오류 발생. 검색 문서의 신뢰도 검증 레이어 또는 키워드 기반 보조 검색 추가 고려.
3. **직접성**: 단답형 질문에는 간결하게 핵심만 답하도록 프롬프트 조정 필요.

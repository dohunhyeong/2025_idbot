# [포트폴리오] RAG 기반 법정감염병 안내 AI 챗봇 서비스 (IDBOT)

## 1. 프로젝트 배경 및 목적
최근 LLM(Large Language Model) 기술의 비약적인 발전으로 다양한 산업 분야에서 인공지능 도입이 가속화되고 있습니다. 그러나 보건 의료 분야에서는 전문적인 도메인 지식의 정확성과 할루시네이션(환각 현상) 문제가 도입의 큰 장벽이 되어왔습니다. 

부산광역시 감염병관리지원단은 매년 '법정감염병 알아보기' 책자를 발간하여 전국의 의료기관 및 유관기관에 배포하고 있습니다. 이 책자는 감염병의 개요부터 진단, 신고 절차, 치료법까지 방대한 정보를 담고 있으나, 긴급한 현장에서 필요한 정보를 즉각적으로 찾아내기 어렵다는 접근성의 한계가 있었습니다. 

본 프로젝트는 이러한 문제를 해결하기 위해, 부산시의 요구사항에 맞춰 감염병 도메인 지식을 정확히 반영하고 답변의 근거(출처)를 명확히 제시할 수 있는 RAG(Retrieval-Augmented Generation) 기반의 AI 챗봇 'IDBOT'을 개발하는 것을 목적으로 진행되었습니다.

## 2. 이해관계자 및 프로그램 요구사항
- **발주처**: 부산광역시 감염병관리지원단
- **주요 요구사항**: 
    1. 144종의 법정감염병 최신 지침서 내용을 100% 반영할 것.
    2. 답변 시 반드시 공식 문서의 출처(URL)를 표기하여 의료 현장에서의 신뢰성을 확보할 것.
    3. 할루시네이션을 방지하기 위해 외부 학습 지식이 아닌, 제공된 문서 데이터 범위 내에서만 답변을 생성할 것.

## 3. 프로젝트 팀 구성 및 역할
- **팀 구성**: 대학원생 2명, 학부생 5명
- **본인 역할 (Full-Stack Developer)**:
    - 프로젝트 전반의 아키텍처 설계 및 RAG 파이프라인(LangChain) 구축
    - FastAPI를 활용한 비동기 백엔드 API 서버 개발 및 MongoDB 연동
    - Docker를 이용한 컨테이너화 및 서버 배포 관리
    - 프론트엔드 인터페이스 구현 및 사용자 경험(UX) 최적화
- **협업 포인트**: 대학원생 팀원과 함께 프로젝트의 기술 방향성을 정의하고 전체 일정을 관리하였습니다. 또한 5명의 학부생 팀원들이 1,440개의 섹션 데이터를 정확하게 HTML로 변환하고 검수할 수 있도록 데이터 구조 가이드라인을 작성하고 프로세스를 리딩하였습니다.

## 4. 상세 기술 스택 및 구현 내용

### 4-1. 데이터 전처리 및 구조화 (Data Engineering)
RAG 모델이 수천 페이지에 달하는 지침서에서 정확한 정보를 찾기 위해서는 데이터의 '원자화'가 필수적이었습니다. 144종의 질병 데이터를 질병개요, 증상, 치료 등 10개 카테고리로 세분화하여 총 1,440개의 독립적인 HTML 문서로 분절하였습니다. 각 문서는 Netlify를 통해 개별 URL을 부여받았으며, 이는 챗봇이 답변 시 사용자에게 직접적인 웹 링크 출처를 제공하는 기반이 되었습니다.

### 4-2. RAG 파이프라인 구축 (LangChain)
- **Document Loading**: `WebBaseLoader`를 활용해 호스팅된 문서를 로드했습니다. 로드 시 메타데이터에 자동으로 포함되는 `source` 정보를 활용해 답변의 출처 기능을 구현했습니다.
- **Embedding**: `OpenAI text-embedding-3-small` 모델을 선택했습니다. 이 모델은 다국어 성능 벤치마크(KoMTEB 등)에서 한국어 및 전문 용어에 대한 의미적 유사성 파악 능력이 우수하며, 이전 모델 대비 비용 효율성이 뛰어나 1,440개의 대규모 데이터를 처리하기에 최적의 선택지였습니다.
- **Vector DB**: `FAISS` 라이브러리를 사용했습니다. 데이터 규모가 서버의 메모리 내에서 충분히 처리가 가능했기에, 클라우드 기반 Vector DB 운영 비용을 절감하면서도 밀리초(ms) 단위의 빠른 검색 속도를 확보할 수 있었습니다.

### 4-3. 백엔드 및 인프라
- **FastAPI**: 실시간 채팅 서비스의 특성을 고려하여 비동기 처리에 최적화된 FastAPI를 사용해 API 서버를 구축했습니다.
- **MongoDB**: 사용자의 질문 이력과 시스템 응답 로그를 저장하기 위해 NoSQL인 MongoDB를 도입했습니다. 스키마가 자유로운 NoSQL의 특징을 활용해 대화 세션 관리와 향후 고도화를 위한 피드백 데이터를 유연하게 수집하고 있습니다.
- **Docker**: 개발 환경과 운영 환경의 일치를 위해 전체 애플리케이션을 Docker 컨테이너로 빌드하여 배포 효율성을 높였습니다.

## 5. 프롬프트 엔지니어링 (Prompt Engineering)
의료 정보의 엄격함을 유지하기 위해 다음과 같은 시스템 프롬프트 전략을 수립하였습니다.
- **검색 기반 답변**: "반드시 Retriever에 검색된 문서만을 활용할 것"이라는 강한 제약을 두어 외부 지식 혼입을 차단했습니다.
- **포맷 지정**: Markdown 형식을 활용해 답변의 가독성을 높이고, `References` 섹션에 URL을 콤마(,)로 구분하여 명시하도록 지시했습니다.
- **방어적 태도**: 적절한 답변을 찾지 못했을 경우 억지로 답하지 않고 "잘 모르겠습니다"라고 답변하게 하여 정보 왜곡을 방지했습니다.

## 6. 프로젝트 결과 및 성과
- **실제 서비스 배포**: 현재 `http://idbot.or.kr` 주소로 정식 배포되어 누구나 사용 가능한 상태입니다.
- **성과 지표**: 2025년 2월, 부산광역시 감염병관리지원단 주관 발표회에서 공식 시연을 마쳤으며, 현재까지 **누적 사용량 3,000건**을 돌파하며 실질적인 정보 제공 도구로서의 가치를 인정받았습니다.
- 단순 LLM(ChatGPT) 대비 RAG구조를 적용한 idiot이 답변유사도(Answer relevancy)를 65% 에서 75%로 약 10% 상승시킴.

## 7. 기술적 회고 및 한계점
- **데이터 로딩 최적화**: `WebBaseLoader` 사용 시 지침서의 핵심인 '표(Table)' 구조가 텍스트 형태로 평면화되어 이를 Context로 활용하여 답변을 생성하니 응답 품질이 저해되는 문제를 발견했습니다. 이를 해결하기 위해 추후 표 구조를 유지하는 파싱 로직을 도입할 예정입니다.
- **검색 정확도 개선**: '신종플루'와 '신종인플루엔자 A'와 같은 동의어 처리에서 검색 엔진이 혼동을 겪는 사례가 확인되었습니다. 향후 쿼리 확장(Query Expansion) 기법이나 도메인 특화 유의어 사전(Thesaurus)을 구축하여 검색 성능을 고도화할 계획입니다.
